## 1. generate_dataset

### 1.0 GenerationConfig

Configuration for the generation pipeline.

| Field | Type | Default | Constraints |
|-------|------|---------|-------------|
| `templates_dir` | string | — | Path to YAML template directory |
| `total_count` | integer | 20000 | > 0 |
| `template_fraction` | float | 0.5 | 0.0 < x < 1.0; fraction of non-hard-negative examples from templates |
| `hard_negative_ratio` | float | 0.10 | 0.0 <= x < 1.0 |
| `seed` | integer \| None | None | Random seed for reproducibility |
| `model` | string | "qwen2.5:3b" | Ollama model tag |
| `ollama_base_url` | string | "http://localhost:11434" | Ollama API base URL |
| `output_dir` | string | "data/" | Writable directory for output files |

### 1.1 generate_dataset

```
generate_dataset(config: GenerationConfig) -> Dataset
```

Top-level function that orchestrates all five stages and produces the final dataset.

**Pipeline order:**
1. Stage 1: `generate_from_templates` → `list[RawExample]`
2. Stage 2: `generate_from_llm` → `list[RawExample]`
3. Orchestrator assigns splits (80:10:10) and IDs to combined Stage 1+2 output → `list[Example]`
4. Stage 3: `validate_labels` on LLM-sourced examples only → `list[Example]` (template-sourced examples pass through unchanged)
5. Stage 4: `inject_hard_negatives` → `list[Example]` (per-split injection)
6. Stage 5: `select_review_sample` → `list[Example]` (sample IDs for human review)
7. `write_dataset` → JSONL files

**REQUIRES:**
- `config` is a valid GenerationConfig.

**ENSURES:**
- Returns a Dataset containing `config.total_count` Example records.
- Split distribution: 80% train, 10% validation, 10% test (rounded to integers).
- Each split contains exactly `config.hard_negative_ratio` hard negatives (rounded to integers).
- All Example records satisfy the Example invariants from entities.md Section 4.
- No Example contains real personal data.
- All four domains (`medical`, `scheduling`, `workplace`, `personal`) are represented.
- All eight SpanLabel values appear across the non-hard-negative examples.

## 2. Stage 1 — Template-Based Generation

### 2.0 RawExample

A pre-split record produced by Stages 1–4 before the orchestrator assigns `id` and `split`.

**Fields:**

| Field | Type | Constraints |
|-------|------|-------------|
| `text` | string | Non-empty |
| `labels` | set\<SpanLabel\> | May be empty; each value must be a valid SpanLabel |
| `risk` | RiskLevel | Valid RiskLevel |
| `rationale` | string | May be empty |
| `is_hard_negative` | boolean | |
| `domain` | string | One of: `medical`, `scheduling`, `workplace`, `personal` |
| `source` | string | One of: `template`, `llm-augmented`, `hard-negative` |

**Contract:**

Identical to Example (entities.md §4) except:
- No `id` field.
- No `split` field.

All DetectionResult invariants and hard-negative invariants still apply.

The orchestrator (generate_dataset, §1) converts each RawExample into a final Example after shuffling and assigning splits and IDs.

### 2.1 generate_from_templates

```
generate_from_templates(templates_dir: string, count: integer, seed: integer | None) -> list<RawExample>
```

**REQUIRES:**
- `templates_dir` contains YAML template files for each domain.
- `count` > 0.

**ENSURES:**
- Returns `count` RawExample records, each with `source: template`, `is_hard_negative: false`.
- Each RawExample's `text` is generated by filling template slots with Faker-generated synthetic values.
- Each RawExample's `labels` set contains the SpanLabel values corresponding to the template's annotated categories.
- Domains are distributed across examples (not all from one template file).
- When `seed` is provided, the output is deterministic for the same seed value.

**MAINTAINS:**
- No template slot is filled with real personal data. Faker generates all values.

### 2.2 Template Format

Each YAML template defines:

```yaml
domain: "medical"
patterns:
  - text: "I saw my {SPECIALTY} at {HOSPITAL} last {DAY_OF_WEEK}"
    labels:
      - "MEDICAL-CONTEXT"
      - "WORKPLACE"
      - "ROUTINE"
    risk: "MEDIUM"
    rationale_template: "Medical specialty and specific hospital narrow identification."
```

**REQUIRES:**
- Each template has a `domain` field matching one of the four valid domains.
- Each pattern has at least one label annotation.
- Each label is a valid SpanLabel.

**ENSURES:**
- After slot filling, the Example's `labels` set contains the template's annotated labels.
- The `rationale` is generated from the `rationale_template` or left to Stage 3 (auto-labeling).

## 3. Stage 2 — LLM-Augmented Generation

### 3.1 generate_from_llm

```
generate_from_llm(count: integer, model: string, seed: integer | None) -> list<RawExample>
```

**REQUIRES:**
- `count` > 0.
- `model` is a valid Ollama model tag (e.g., `"qwen2.5:3b"`).
- Ollama is running and the model is available.

**ENSURES:**
- Returns `count` RawExample records, each with `source: llm-augmented`, `is_hard_negative: false`.
- Each RawExample's `text` reads as natural, diverse free text (not templated).
- Each RawExample includes the set of SpanLabel categories present.
- Examples are distributed across all four domains.
- All eight SpanLabel values appear across the returned examples.

**MAINTAINS:**
- The generation prompt instructs the LLM to produce synthetic content only. No real personal data is solicited.
- LLM calls use the Ollama `/api/chat` endpoint with `format: "json"`.

---

## 3.2 LLM Call Resilience (applies to Stages 2, 3, and 4)

All stages that call the Ollama API must handle transient failures gracefully:

1. **Empty or unparseable responses.** The model may return empty content or malformed JSON despite `format: "json"`. The caller must strip markdown code fences and treat empty or unparseable content as a failed batch, not an unrecoverable error.
2. **Per-batch error handling.** Each batch call must be wrapped in error handling that catches `RuntimeError` and `JSONDecodeError`. A failed batch must be logged and skipped, not propagate as an unhandled exception.
3. **Retry with bounded attempts.** After a failed batch, the stage may retry. After 5 consecutive failed batches with no progress, the stage raises `RuntimeError`.
4. **Partial progress.** Stages must preserve examples from successful batches even if later batches fail.

## 4. Stage 3 — Auto-Labeling Validation

### 4.1 validate_labels

```
validate_labels(examples: list<Example>, model: string) -> list<Example>
```

**REQUIRES:**
- `examples` is a list of Example records (with splits and IDs assigned).
- `model` is a valid Ollama model tag.
- Ollama is running and the model is available.

**ENSURES:**
- Only examples with `source: llm-augmented` are sent to the LLM for validation. Examples with `source: template` are passed through unchanged.
- For each validated Example, the `labels`, `risk`, and `rationale` fields have been verified or corrected by a second LLM pass.
- Any incorrect label is corrected or removed. Missing labels are added.
- RiskLevel is validated against the combination of labels.
- Rationale is generated for examples with risk MEDIUM or HIGH.
- Examples where the validation pass disagrees with the original and cannot be resolved are removed (and must be regenerated to meet count targets).
- Returns all examples: template-sourced (unchanged) + validated LLM-sourced.

**Response matching:** The validation response is an array whose entries correspond to the input batch. Each entry includes an `id` field. The parser matches response entries to input examples as follows:
1. **ID match (primary):** Look up the entry's `id` in the input batch. If found, apply the validation result to that example.
2. **Positional fallback:** If the response array has the same length as the input batch and every `id` in the response fails to match any input example, match entries by position (response index `i` maps to input index `i`).
3. **Unmatched entries:** Response entries that match neither by ID nor by position are logged and discarded. Input examples with no matching response entry are removed (treated as unresolvable).

**MAINTAINS:**
- LLM calls use the Ollama `/api/chat` endpoint with `format: "json"`.

## 5. Stage 4 — Hard Negative Injection

### 5.1 inject_hard_negatives

```
inject_hard_negatives(examples: list<Example>, ratio: float, model: string) -> list<Example>
```

**REQUIRES:**
- `examples` is the validated list from Stage 3, with splits and IDs assigned.
- `ratio` is in (0.0, 1.0).
- `model` is a valid Ollama model tag.
- Ollama is running and the model is available.

**ENSURES:**
- Returns `examples` plus additional hard negative Example records.
- For each split, hard negatives are injected so that hard_negative_count / split_total_after_injection = `ratio` (rounded to integer).
- Each hard negative has: `is_hard_negative: true`, `labels: set()`, `risk: LOW`, `rationale: ""`, `source: hard-negative`.
- Hard negative texts are generated via Ollama and contain location, time, organization, or person references that are not PII in context.
- Hard negative IDs follow the `{split}-{zero-padded-number}` pattern, numbered after the existing examples in each split.

**MAINTAINS:**
- LLM calls use the Ollama `/api/chat` endpoint with `format: "json"`.

---

## 6. Stage 5 — Human Review

### 6.1 select_review_sample

```
select_review_sample(dataset: Dataset, ratio: float) -> list<Example>
```

**REQUIRES:**
- `ratio` is 0.01 (1%).
- `dataset` contains all examples across all splits.

**ENSURES:**
- Returns a random sample of `ratio` * len(dataset) examples.
- The sample is drawn proportionally from each split.
- Selection is random (not stratified by domain, label, or difficulty).

### 6.2 apply_corrections

```
apply_corrections(dataset: Dataset, corrections: list<Correction>) -> Dataset
```

**REQUIRES:**
- Each Correction references an Example by `id` and contains corrected field values.

**ENSURES:**
- Corrected fields are applied to the matching Example records.
- No new Examples are added.
- No Examples are removed.
- Split assignments are not changed.
- All corrected Examples still satisfy Example invariants from entities.md Section 4.

---

## 7. Output Format

### 7.1 write_dataset

```
write_dataset(examples: list<Example>, output_dir: string) -> void
```

**REQUIRES:**
- `examples` is a non-empty list of valid Example records.
- `output_dir` is a writable directory path.
- Every Example has a valid `split` field.

**ENSURES:**
- Writes three files: `train.jsonl`, `validation.jsonl`, `test.jsonl`.
- Each file contains one JSON-serialized Example record per line.
- Each line is valid JSON and deserializes to a valid Example.
- Examples are grouped by their `split` field into the corresponding file.

### 7.2 read_dataset

```
read_dataset(input_dir: string) -> list<Example>
```

**REQUIRES:**
- `input_dir` contains at least one of `train.jsonl`, `validation.jsonl`, `test.jsonl`.

**ENSURES:**
- Returns Example records deserialized from all JSONL files present.
- Each record is validated on load (Example invariants hold).
- Records are returned in file order: train, then validation, then test.
